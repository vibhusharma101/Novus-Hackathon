## Pass 1 — OWASP Top 10

1. **Broken access control** — `ExchangePage` calls `auth()` + `redirect('/sign-in')` before any DB access. `supabaseAdmin` is only used server-side (SSR module boundary, never sent to the browser). The admin query selects only `listings` + public recycler fields; no tenant-scoped or user-private data is fetched via admin path. Liabilities fetch uses `createUserClient()` (RLS-scoped). ✅
2. **Cryptographic failures** — No secrets or tokens appear in code or logs; `SUPABASE_SERVICE_ROLE_KEY` is consumed via env var in the existing `supabase.ts` module, not in the new files. ✅
3. **Injection** — All DB queries use parameterised Supabase JS client. `listing.id` used in `router.push(…)` is a UUID read from DB state, not from user input; Next.js router does not evaluate it as code. ✅
4. **Insecure design** — No auth bypass surface. Realtime `postgres_changes` subscription carries the Clerk JWT (set in `getSupabaseBrowserClient(token)`); Supabase evaluates the `"listings: read active"` RLS policy against the subscriber's JWT, so only authenticated users receive events. ✅
5. **Security misconfiguration** — No debug flags, no CORS changes, no verbose error strings exposed to the browser. The `try/catch` in the page swallows `createUserClient` errors silently (user lands on exchange with 0 liabilities); no stack trace or internal error leaks to the client. ✅
6. **Vulnerable dependencies** — No new packages in this diff; existing packages are pinned with `^` semver (same as prior commits). ✅
7. **Auth/identification failures** — Auth check is at page entry; client component depends on the existing Clerk session via `useAuth()`. No new login/session management surface introduced. ✅
8. **Software/data integrity** — No deserialization of untrusted blobs. Realtime `payload.new` is typed as `Listing` (a TS cast, not a parse), but the data originates from Supabase's own CDC pipeline over an authenticated channel — not from user-controlled input. ✅
9. **Logging/monitoring failures** — No `console.log`, no PII logged. ✅
10. **SSRF** — No server-side URL fetches. ✅

### `OrderBook` realtime subscription (client)

| Threat | Assessment |
|---|---|
| Spoofing | Clerk JWT sent as `Authorization` header on the Supabase client; Supabase validates it server-side before accepting the channel. ✅ |
| Tampering | Client state (`listings`) is populated from server-pushed events, not from user-editable fields. ✅ |
| Repudiation | No write action on this screen (Buy navigates to checkout, which is B6). ✅ |
| Info disclosure | On INSERT, the component fetches the recycler row via `sb.from('recyclers').select(…)` using the user JWT → `"recyclers: public read"` policy gates this to authenticated callers and returns only public columns (no PII). ✅ |
| DoS | A malicious authenticated recycler could rapidly INSERT hundreds of listings causing the client to fire a recycler-fetch per INSERT. Each event triggers one SELECT; the browser and Supabase connection pool bound the blast radius. Acceptable risk for a hackathon demo. LOW. |
| EoP | Realtime subscription is read-only; no privilege escalation path. ✅ |

### LOW

- `src/app/(dashboard)/dashboard/exchange/page.tsx:16–20` — **No row LIMIT on `supabaseAdmin` listing fetch.** An authenticated recycler could publish thousands of active listings; the SSR fetch would materialise the full result set on every page load and send it all in the initial HTML payload. Attack: authenticated recycler spams listings → every buyer page-load fetches and serialises them all → memory/latency spike. Fix: add `.limit(200)` (or a sane cap); the client already paginates at 10.

- `src/components/exchange/order-book.tsx:147–163` — **One realtime recycler SELECT per INSERT event.** If a burst of new listings arrives simultaneously (e.g., a seller bulk-publishes), the client fires one authenticated Supabase SELECT per event with no debounce or deduplication. Not exploitable today (recycler publishes one listing at a time in the demo flow), but worth noting for Phase 3.

## Verdict

**CLEAR** — 0 HIGH, 0 MEDIUM. Two LOW items flagged for awareness; neither is exploitable in the current demo flow.
