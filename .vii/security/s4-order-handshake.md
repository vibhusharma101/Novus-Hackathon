The page additionally double-scopes `.eq('recycler_id', recycler.id)`. A recycler can only accept/decline **their own** orders ‚Äî no IDOR. ‚úÖ
2. **Cryptographic failures** ‚Äî `provisionalCertificateId` is a deterministic display reference, not a secret/credential (already noted in B7). ‚úÖ

## Findings

### HIGH
_None._

### MEDIUM ‚Äî RESOLVED in this commit

- `supabase/schema.sql` (`certificates: system inserts`) ‚Äî **The certificate insert policy was `with check (true)`, letting any authenticated user insert a certificate row for an arbitrary `order_id`.** S4 makes this reachable: it issues certs via the user client and treats a duplicate (`23505`) as already-issued.
  - **Attack (closed):** an attacker who knows a victim's order UUID pre-inserts `certificates {order_id: victimOrder, reference_id: attacker-chosen}`. When the legitimate recycler accepts, the action's insert hits `23505`, is treated as success, and the issued certificate carries the **attacker's** reference_id (integrity/griefing). Requires knowing the random order UUID (not enumerable), hence MEDIUM not HIGH.
  - **Fix applied:** replaced the policy with `"certificates: recycler of order inserts"` ‚Äî `with check (order_id in (orders where recycler_id ‚àà own recyclers))`. `respondToOrder` inserts as the recycler fulfilling the order, so the legitimate path is unaffected; third-party pre-inserts are now rejected by RLS.

### LOW

- `src/lib/actions/seller.ts` (`respondToOrder`) ‚Äî if the cert insert fails with a non-`23505` error after the status update committed, the action returns `{ ok: false }` while the order is already `transferred` (resolves on reload). Honest, not swallowed; cosmetic. (Also flagged MINOR by `/vii-review`.)

## Verdict

**CLEAR** ‚Äî 0 HIGH, 0 unresolved MEDIUM. The accept/decline path is access-control-clean and race-safe; the one MEDIUM (over-permissive certificate insert policy, made reachable by S4) was **fixed in the same commit** by scoping inserts to the order's recycler.
