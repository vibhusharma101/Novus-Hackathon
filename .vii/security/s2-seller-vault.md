## Pass 1 — OWASP Top 10

1. **Broken access control** — `SellerVaultPage` gates on `auth()`, resolves the caller's own `recyclers` row via RLS `.single()`, then fetches listings/orders with an explicit `.eq('recycler_id', recycler.id)` **on top of** RLS. The new `"listings: recycler reads own"` policy is scoped to `recycler_id in (own recyclers)` — a recycler can read only their own listings, no broader exposure. No user-controlled ID reaches a query unscoped. No IDOR. ✅
2. **Cryptographic failures** — no secrets/keys; `createUserClient` only, no `supabaseAdmin` in S2. ✅
3. **Injection** — realtime filter `recycler_id=eq.${recyclerId}` interpolates a **server-derived UUID** (from the authenticated session's recycler row), not user input; parameterised inserts elsewhere. No SQL/PostgREST injection vector. ✅
4. **Insecure design** — **Denormalization review:** `buyer_gstin`/`buyer_company_name` are written by `placeOrder` from the buyer's **own RLS-scoped `brands` row** (`brand.gstin`, `brand.company_name`) — never from client input, so a buyer cannot forge another entity's GSTIN onto an order. Exposure of these fields is confined by the existing `"orders: buyer or recycler"` policy to exactly the buyer and the one involved recycler — the precise audience that needs the GSTIN for the CPCB transfer. Unlike the recyclers `public read` policy (fixed in S1), the orders policy is **not** a broad public-read, so no column-level lockdown is required here. ✅
5. **Security misconfiguration** — redirects on missing recycler (`/onboarding/seller`); no verbose errors. ✅
6. **Vulnerable dependencies** — none added. ✅
7. **Auth/identification failures** — page gated; `placeOrder` already gated. ✅
8. **Software/data integrity** — realtime `payload.new` originates from the Postgres WAL over an authenticated, RLS-filtered channel — not client-injectable. The seller cannot receive an order event for a row they don't own (RLS + filter). ✅
9. **Logging/monitoring failures** — no `console.log` of PII/secrets. ✅
10. **SSRF** — none. ✅

## Pass 2 — STRIDE (seller-vault realtime + placeOrder change)

### LOW

- `src/app/seller/vault/page.tsx:24` — **No `LIMIT` on the own-listings / own-orders fetches.** A seller with very many listings/orders loads them all on each render (and all orders feed the client memo). Bounded by the seller's own data and not cross-tenant, so low impact; add `.limit()` + pagination when volume grows. (Same class as the B5 SSR note.)

- `supabase/schema.sql` — **`buyer_gstin` is now stored in a second location (`orders`) in addition to `brands`.** A broader PII storage footprint, but confined by RLS to the buyer + involved recycler and justified by the transfer workflow. Noted for data-minimization awareness; no action needed for the demo.

## Verdict

**CLEAR** — 0 HIGH, 0 MEDIUM. The notable design change (denormalizing buyer GSTIN onto orders) is correctly sourced server-side from the buyer's own row and correctly scoped by the existing buyer-or-recycler order policy. Two LOWs noted for scale/data-minimization.
