## Pass 1 — OWASP Top 10

1. **Broken access control** — `placeOrder` calls `auth()` before any DB access. `buyer_id` is derived from an RLS-scoped `.from('brands').select('id').single()` query — never from client input. `recycler_id`, `price_per_kg`, `category`, `listing_id` are all taken from the server-side `supabaseAdmin` listing fetch, so a buyer cannot manipulate price or route the order to an arbitrary recycler. `CheckoutPage` calls `auth()` + redirects. ✅
2. **Cryptographic failures** — No secrets or tokens logged. `orderId` (UUID) returned to client is Supabase-generated. ✅
3. **Injection** — All DB queries use parameterised Supabase JS client. `listingId` reaches `.eq('id', listingId)` which is parameterised. `qty_kg` is validated as a finite positive number before use. No shell exec, no template injection. ✅
4. **Insecure design** — See MEDIUM below. No auth bypass path detected.
5. **Security misconfiguration** — Error messages are generic (`'Could not place order. Please try again.'`). The one dynamic error `Quantity exceeds available volume (${listing.qty_kg} kg)` leaks qty, but `qty_kg` is already publicly visible in the order book. ✅
6. **Vulnerable dependencies** — No new packages. ✅
7. **Auth/identification failures** — `auth()` is the first call in `placeOrder` and `CheckoutPage`. No new login/session surface. ✅
8. **Software/data integrity** — `price_per_kg`, `credits_cost`, `platform_fee`, `total` all computed server-side from the admin-fetched listing. A buyer cannot submit a manipulated price — the form's price display is read-only and the server ignores it entirely. `expires_at` computed server-side. ✅
9. **Logging/monitoring failures** — No `console.log` of PII, tokens, or DB errors. ✅
10. **SSRF** — No server-side URL fetches driven by user input. ✅

## Findings

### HIGH
_None._

### MEDIUM

- `src/lib/actions/buyer.ts:134–175` — **No listing quantity reservation and no per-call rate limit on `placeOrder`.** The action creates an `orders` row but does not decrement `listing.qty_kg` or update `listing.status`. Two authenticated buyers racing to call `placeOrder` on the same `listingId` both receive `ok: true`; the listing remains `active`; a single buyer can also loop-call `placeOrder` to fill the `orders` table with uncapped `pending` rows against the same listing.
  - **Attack 1 (oversell):** Two buyers race to buy the full `qty_kg` of one listing → both orders created → seller receives two simultaneous handshakes for credits that only exist once.
  - **Attack 2 (DoS):** Authenticated buyer scripts repeated `placeOrder` calls → `orders` table balloons with pending rows; no uniqueness constraint prevents it.
  - **Fix (hackathon-acceptable):** Add `unique(listing_id, buyer_id)` constraint on `orders` for `status = 'pending'` (DB-level guard against duplicate orders per buyer), and/or atomically decrement `listings.qty_kg` and flip status in a DB function called from `placeOrder`.

### LOW

- `src/app/(dashboard)/dashboard/checkout/[listingId]/page.tsx:20–25` — `supabaseAdmin` SSR listing fetch has no row LIMIT. Same as B5 note: with many active listings the call is unbounded, though here it is filtered to a single `id` so the result is at most one row. ✅ (effectively bounded by the UUID filter — this is a non-issue here, noting for completeness only.)

## Verdict

**FIX-MEDIUM-RECOMMENDED** — 0 HIGH, 1 MEDIUM. The MEDIUM (no listing reservation + no rate limit) is a known hackathon trade-off for the single-seller demo flow, but should be noted before a real launch.
