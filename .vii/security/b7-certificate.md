## Pass 1 — OWASP Top 10

1. **Broken access control** — The `[id]` route param is user-controlled. Defence is layered: (a) `auth()` first; (b) `brand` resolved via RLS `.single()` (only the caller's brand); (c) order fetched with **both** `.eq('id', orderId)` **and** `.eq('buyer_id', brand.id)` **plus** the `"orders: buyer or recycler"` RLS policy. An attacker substituting another order's UUID gets `null` → `redirect('/dashboard/exchange')`. The explicit `buyer_id` filter also prevents the *recycler* party (who can also read the order under RLS) from loading the buyer's certificate. No IDOR. ✅
2. **Cryptographic failures** — No secrets, tokens, or service-role key in the new files. The "Hash:" line (`shortHash`) is a cosmetic non-cryptographic display string, clearly decorative, not used for integrity. ✅
3. **Injection** — All queries parameterised via Supabase JS. `orderId` flows into `.eq('id', ...)` (parameterised). React auto-escapes all rendered strings (company name, GSTIN, reg no). No `dangerouslySetInnerHTML`. ✅
4. **Insecure design** — Provisional-cert semantics are explicit; no auth-bypass. No state mutation on this page (read-only), so no race/TOCTOU surface. ✅
5. **Security misconfiguration** — No verbose errors; failures `redirect` to safe routes. No debug flags. ✅
6. **Vulnerable dependencies** — None added. ✅
7. **Auth/identification failures** — `auth()` gates the page; no new auth surface. ✅
8. **Software/data integrity** — No deserialization of untrusted data. Certificate values come from RLS-scoped DB rows the caller owns. ✅
9. **Logging/monitoring failures** — No `console.log` of PII/tokens in the diff. GSTIN is rendered to the page but only to its owner (the buyer). ✅
10. **SSRF** — No server-side URL fetch driven by user input. ✅
