# Security audit: Phase 4 — AI endpoint rate limiting (CSO fix)

_Generated by /vii-cso on 2026-06-13. Base: main._

## Surface
- New handlers: rate-limit guard added to `/api/copilot`, `/api/estimate-liability`, `/api/chat`
- New DB objects: `ai_rate_limits` table (RLS-locked) + `check_ai_rate_limit(text, int, int)` function (service-role-only EXECUTE)
- New helper: `src/lib/rate-limit.ts` (`checkRateLimit`, via `supabaseAdmin.rpc`)
- New external calls: none (one extra Postgres RPC per AI request)
- New env vars / deps: none

## Pass 1 — OWASP Top 10

1. **Broken access control** — The rate-limit key is the session `userId` from `auth()`, so a user cannot target another user's bucket. The `check_ai_rate_limit` function has EXECUTE **revoked from public/anon/authenticated and granted only to `service_role`**, so an authenticated user cannot invoke it directly via PostgREST RPC. The `ai_rate_limits` table has RLS enabled with no policies → API roles see/insert no rows; `service_role` (server-only) bypasses RLS. ✅
2. **Cryptographic failures** — no secrets. ✅
3. **Injection** — RPC args are bound parameters via `supabaseAdmin.rpc(...)`; the PL/pgSQL function uses them directly (no dynamic SQL / `EXECUTE`). `p_user` is the server-derived session id; `p_limit`/`p_window_seconds` are server constants. No injection vector. ✅
4. **Insecure design / race** — the counter increment is atomic: `insert … on conflict … do update set count = count + 1` takes a row lock, so concurrent requests serialize correctly (no lost updates / TOCTOU). Fixed-window allows a ≤2× burst at a window boundary — acceptable for a cost-DoS control (sustained rate stays bounded). ✅
5. **Security misconfiguration** — generic `429` message; RPC locked down; table RLS-locked. ✅
6. **Vulnerable dependencies** — none. ✅
7. **Auth/identification failures** — limiter runs after `auth()`, keyed to the authenticated user. ✅
8. **Software/data integrity** — no untrusted deserialization. ✅
9. **Logging/monitoring failures** — none. ✅
10. **SSRF** — none. ✅

## Pass 2 — STRIDE on the limiter

| Threat | Assessment |
|---|---|
| Spoofing | Key is the session `userId`; can't be spoofed to another bucket. ✅ |
| Tampering | Table RLS-locked, RPC `service_role`-only — authenticated users can't read or alter counts. ✅ |
| Repudiation | N/A. |
| Info disclosure | Counter table holds no sensitive data and is RLS-locked regardless. ✅ |
| DoS | This **is** the DoS mitigation; it adds one cheap RPC per request and rejects (`429`) before any LLM call. **Fail-open** on RPC error is not attacker-triggerable (args are server-set; errors are infra-only), so it can't be used to bypass. ✅ |
| Elevation of privilege | RPC EXECUTE confined to `service_role`; no escalation. ✅ |

## Findings

### HIGH
_None._

### MEDIUM
_None — the previously-flagged unthrottled-LLM-endpoint MEDIUM (see `p4-copilot.md`, `p4-estimator.md`) is **RESOLVED**: all three AI routes now reject with `429` once a user exceeds 30 requests / 60s, enforced by a shared atomic Postgres counter._

### LOW

- `src/lib/rate-limit.ts` — **Fail-open on RPC error.** If the migration hasn't been applied (or the DB is unreachable), `checkRateLimit` returns `true` (allowed), reverting to the prior unthrottled behavior. Deliberate (availability > strict enforcement for a best-effort limiter) and **not attacker-controllable** — RPC args are server-set, so an attacker can't force the error path. Apply `supabase/schema.sql` to activate enforcement.
- `supabase/schema.sql` — `ai_rate_limits` rows are never pruned (one per user per window) → unbounded growth over time. Availability/storage, not security; add a cron prune for production (also flagged by `/vii-review`).

## Verdict

**CLEAR** — 0 HIGH, 0 MEDIUM. The rate limiter resolves the prior MEDIUM across all three AI endpoints with a shared, atomic, server-only Postgres counter; the new RPC surface is correctly locked to the service role. Two LOW notes (fail-open activation requirement, counter-row pruning) are operational, not exploitable.
