### LOW

- `src/lib/actions/seller.ts` (`createListing`) ‚Äî **No per-seller cap or rate limit on listing creation.** An authenticated (self-asserted "verified", per S1 LOW) seller can publish unlimited active listings, bloating buyer order books and the `listings` realtime stream ‚Äî compounded by B5's unbounded order-book fetch. No data exposure; impact is UX/throughput degradation. Hardening: cap active listings per recycler and/or rate-limit the action before production.

- `src/app/seller/listings/new/page.tsx:36` ‚Äî **Unbounded read of all active listings** for the pricing assistant (`Math.min(...prices)` spread). Bounded by public data and fine at demo scale; aggregate server-side (or `.limit()`) at high volume. (Same class as the B5 SSR note; also flagged MINOR by `/vii-review`.)

## Verdict

**CLEAR** ‚Äî 0 HIGH, 0 MEDIUM. `createListing` is access-control-clean (recycler_id server-derived, RLS `with check` backstop). Two LOWs are scale/abuse-hardening items for production.
